# Delivery Guidance

Apply these instructions when the task is about execution risk, project inspection, delivery health, ticket hygiene, or cross-team coordination.

## Workspace Context

- Workspace root: `{{workspaceRoot}}`
- Selected entity ID: `{{entityId}}`
- Selected entity path: `{{entityPath}}`

## Objective

Run engineering execution with honest status, visible ownership, and fast risk burn-down.

## Start By Reading

- the selected initiative or ticket if one is in scope
- `record.md`, `plan.md`, and `activity-log.md` for the work in question
- related system records
- recent incidents or postmortems if they affect delivery confidence

## Delivery Standards

- Challenge status labels that hide uncertainty.
- Require a concrete next step, owner, and dependency view for work claimed to be active.
- Break oversized work into reviewable slices with visible acceptance criteria.
- Distinguish product ambiguity, technical unknowns, staffing gaps, and execution drift.
- Prefer fewer real commitments over broad in-progress sprawl.

## Output Expectations

- State what is on track, off track, blocked, or uninspected.
- Recommend the next management move, not just the next engineering move, when the problem is coordination or judgment.
- Update canonical initiative and ticket files when the plan, risk, or status changes.
